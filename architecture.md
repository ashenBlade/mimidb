# Архитектура

## Основные особенности

Первая версия должна поддерживать базовый OLTP функционал СУБД:

- Только простые SELECT запросы: SELECT ... FROM ... WHERE ... (пока без JOIN'ов)
- UPDATE без подзапросов, а прямые выражения для вычисления
- DELETE аналогично - только обычный WHERE
- Без DDL, вся схема назначается статически (системный каталог не нужен)

Ядро:

1. Изоляция - SI+SSI (пока только SI, serializable не надо)
2. Хранилище гибридное - append only (n2o) + undo log
3. COOP VACUUM
4. В таблицах только heap

## Хранение данных

### clog

По аналогии с pg я веду свой clog - commit log, но в отличие от него, мне нужно хранить не только статус, но и то, какой CSN у этой tnx при коммите.

Поэтому clog организован следующим образом - каждая страница 8Кб состоит из 8 байтовых записей.
Каждая запись состоит из полей:

1. 64 бит - aborted
2. 63 бит - committed
3. остальные - CSN или UndoLocation

Таким образом, максимальные CSN и UndoLocation могут использовать только 62 бита, но в реальном мире даже это очень большие цифры.

Кроме того, освободившиеся биты я могу использовать не только здесь, но, например, в кортежах (но для этого и XID тоже надо так ограничить).

Таким образом, интерпретация следующая:

- Если aborted или ничего (т.е. in-progress), то UndoLocation
- Если committed, то CSN

### UNDO лог

Требования:

1. Быстро откатить все действия транзакции - чтобы не искать по всем возможным местам
2. Быстро получить конкретную UNDO запись - чтобы быстро получить предыдущую версию кортежа для UNDO
3. Не выделять гораздо больше места чем требуется - иначе много небольших UPDATE могут быстро все место на диске занять
4. Поддержка неограниченного места в UNDO логе для любой транзакции - чтобы мы не упирались в размер UNDO лога, даже если все хорошо (как у оракла)

Решение:

Весь UNDO лог будет поделен на отдельные батчи по 64К (uint16max) транзакций в каждом, т.е. для определения батча просто делим XID на uint16max (убираем 2 лидирующих байта).

Для большей конкурентности каждый такой чанк может делиться на 256 сегментов (1 байт).

Каждый сегмент имеет строение WAL - все организовано в виде цепочки записей, каждая из которых указывает на предыдущую. Это нужно для того, чтобы после записи мы не могли больше изменить ее, т.е. отматывать ничего не нужно.

Для каждого батча мы будем делать свой control file. В нем будет содержаться разная информация.
В частности:

- Количество сегментов - чтобы не искать по всем файлам
- Адрес последней записи - чтобы отслеживать хвост и уметь итерироваться по нему, для применения UNDO записей

Пока не думал про чекпоинты и как они могут нам помочь

### WAL

Конечно-же мне потребуется WAL.

Но при этом мы добавим возможность использования нескольких WAL. Это позволит добиться большей пропускной способности. На данный момент планирую 2: для самих и UNDO лога.

> Далее, практически описание реализации в PG

Весь WAL будет поделен на сегменты с каким-то размером (каким - неважно).
Когда в текущем заканчивается место, то: если чекпоинт прошел, то переиспользуем какой возможно, либо создаем новый сегмент.

Каждая запись будет стандартной: заголовок, crc, тип записи, сами данные, указатель на предыдущую запись.
Указатель на запись будет просто смещением, как `XLogRecPtr`.

### heap

За основу будет взят обычный HEAP (как в pg), но с учетом моих требований.

Заголовок страницы:

1. LSN - XLogRecPtr, указатель на запись в WAL
2. Checksum - чек-сумма страницы
3. vacated - свободные байты фрагментации
4. lower - начало свободного места
5. upper - конец свободного места

> Практически тоже самое, что и в PG, но только удалил ненужные поля (checksum в принципе мне тоже сейчас не нужен), но добавлю для прикола

Если сравнивать с pg, то нет полей:

- pd_flags - флаги пока не нужны
- pd_special - special пока не нужен
- pd_pagesize_version - версионирование мне не нужно
- pd_prune_xid - у меня пока нет prune и вакуума не будет

Но добавились:

- vacated - количество байт, которые освободились после удаления/обновления, но они между кортежами, поэтому мы не можем их использовать. Если есть, можем дефрагметировать страницу и вставить кортеж.

В отличие от Oracle:

- Единица атомарности кортеж, а не сама страница, поэтому не нужно сохранять SCN на странице
- Все блокировки только в памяти (даже если будет OOM), поэтому ITL нет

### Кортеж

1. XID - транзакция, создавшая этот кортеж
2. CSN - CSN этой tnx
3. UndoLocation - UNDO записи для этого кортежа
4. Флаги:
   1. `deleted` - кортеж удален (не знаю надо ли отличать удален от updated)

CSN используется как хинт. Во время работы (insert/update/delete) tnx еще в работе, поэтому CSN неизвестен.
Но как только мы закоммитились, то нельзя идти и выставлять CSN на этих страницах, так как изменения могли затронуть очень много страниц и это продолжится очень долго.
Это будут выполнить другие траназакции по мере работы - выставлять CSN и обновлять другую информацию (кооперативное обновление).

Замечания:

- UndoLocation можно использовать вместо ctid как в pg, если я обновляю кортеж, то информация о старом и новом должна храниться у меня в Undo записи, поэтому дополнительного поля не нужно
- Во флагах пока нет информации о блокировках, т.к. они будут только в памяти.

## Организация транзакционности

В качестве механизма проверки видимости используется CSN - Commit Sequence Number.
64 бит число, увеличивающееся с каждым коммитом.

В таком ключе проверка видимости сводится к простому сравнению 2 чисел, без использования большого снапшота как в pg - \[xmin,ip,xmax\].
Это ускоряет не только проверку, но и сам момент получения снапшота (нет ботлнека `GetSnapshotData`).

Для уменьшения использования дискового пространства используется механизм UNDO лога - по возможности старые версии переписываем новыми и сразу очищаем место, но если нужно откатить транзакцию или получить более старую версию, то идем в undo лог и восстанавливаем информацию.
Такой подход эффективно можно реализовать с помощью IOT, но для heap приходится искать обходные пути, т.к. после обновления новый кортеж может быть вставлен только на новое место и другие транзакции могут просто не обнаружить этого кортежа, если его сразу уже удалить, - т.е. данные будут потеряны на пустом месте. В IOT этого не случается, т.к. данные упорядочены и после назначения они никуда деться не могут.

Основная проблема в том, что он может требовать большого количества места. Во многих БД для этого существует retention период - undo запись удаляется по истечении этого времени. В моей системе такого не будет, а поведение похоже на pg - отслеживается последний CSN в снапшоте - все меньшие автоматически видны, т.е. заморожены, FREEZE. Но если в PG FREEZE выполнялся жадно, т.к. XID wraparound, то у меня этот FREEZE будет выполняться лениво - во время чтения если обнаружил, что XID'а уже нет (на диске), то значит от закоммитился, но т.к. удалиться он может только если прошел oldestCSN, то значит автоматически такие записи считаю замороженными. Учитывая это можно даже не выставлять CSN как замороженный, т.к. он автоматически будет виден для каждого снапшота.

Для того, чтобы отслеживать состояние транзакции я буду использовать сам UNDO лог - в нем будет храниться состояние транзакции. В частности там будет храниться CSN этой транзакции.
Такой выбор пал, т.к. этот CSN нужно где-то отслеживать для каждой транзакции, но использовать для этого SLRU не вариант, т.к. RO транзакции ничего не пишут и у них нет следов деятельности, соответственно нет необходимости отслеживать это.

Кроме того, когда мы храним CSN в UNDO, то появляется оптимизация локальности данных - если на кортеже CSN не выставлен, то я иду в UNDO буферы и читаю эту страницу, т.к. этот CSN не выставлен, то очень вероятно, что я еще в процессе выполнения, поэтому необходимо будет применить UNDO запись, а это сделать будет легче, т.к. страницу я уже прочитал, осталось только найти нужный блок.

Последний вопрос - как по XID получать его UNDO location (первый блок, который содержит также и ее информацию).

Для этого мы немного модифицируем clog из PG - он будет хранить в себе 8 байт - 2 старших бита - aborted/committed, как в pg, но остальные байты - это либо Undo Location, либо CSN.
Если committed выставлен, то там хранится CSN. В противном случае - undo location.

Также, чтобы не занимать место, мы используем идею postgres и введем понятие virtual xid - если транзакция ничего не меняла, т.е. чисто виртуальная, то для нее будет использоваться этот специальный XID.
Его отличие в том, что на диск он записываться не будет и глобальный XID изменять тоже не будет.
Таким образом, XID нужен только тогда, когда мы что-то меняем.

## Control File

Для БД нужен контрольный файл, в котором будет отслеживаться состояние БД.

На данный момент мне нужно отслеживать:

1. Latest XID - последний использованный XID
2. Latest CSN - последний использованный CSN
3. Latest LSN - адрес последней записи в WAL
4. REDO LSN - адрес начала выполнения чек-поинта

### UNDO/REDO recovery

Я использую UNDO, поэтому записи в WAL не особо важны ЕСЛИ я не закоммитился, но именно поэтому UNDO для меня важен.

Порядок восстановления:

1. REDO - применяю его записи - сначала (последней контрольной точки)
2. UNDO - отменяю записи транзакций, которые были abort/in-progress, - с конца (для каждой tnx с конца применяю записи)

С помощью clog и control file я получаю информацию кто закоммитился/abort/ip. Из этого понимаю, какие записи в REDO применять.
После этого начинаю применять UNDO записи:

- В clog должны быть записаны адреса первых UNDO записей для каждой tnx.
- Из них я узнаю какие сегменты мне надо просмотреть.
- Дохожу до конца, а затем с хвоста в голову иду и применяю записи для незакоммиченных tnx.

Когда я делаю изменения:

1. Всегда создаю UNDO запись первой, только после меняю сами данные
2. После создания этой UNDO записи обязательный `fsync`
3. Далее, делаю запись в WAL, но пока БЕЗ fsync (т.к. незакоммитился)
4. Изменяю данные на странице
5. ...
6. В конце всей транзакции, когда коммичусь - делаю fsync WAL

### Чек-поинт

Чек-поинт реализуется как в pg.

Используется реализация fuzzy checkpoint - весь чекпоинт состоит из 2 точек - начало и конец.
Во время чек-поинта сбрасываются все возможные данные, а не только буферы. Например, CLOG, subtrans, multixact и т.д.

Сейчас я не буду задумываться о восстановлении серьезно. Пока все выключения у меня нормальные и ничего страшного произойти не может.

## Внутренности

### Transaction

Модель выполнения - многопоточная. На каждое соединение выделяю свой поток, в котором будут выполнятся все транзакции.

Для каждой сессии есть свой объект, ее представляющий. Там вся необходимая информация для обработки и чтобы другие сессии могли посмотреть внутрь.

Каждая сессия подключается к конкретной БД и может работать с ней. Пока не планирую добавлять что-то другое.

### Buffer pool

Пул страниц организован как в pg - выделяется огромный массив из страниц.

Для него будет использоваться параллельный массив самих страниц, а также дополнительных элементов, таких как заголовок, флаги, io cond vars и т.д.

## OLTP

### Язык запросов

Сейчас не будет SQL или подобного, а простой интерфейс, похожий на Redis:

- SELECT - получаю все записи
- INSERT v1, v2, v3 - вставляю кортеж с этим значением
- UPDATE fieldno +-value oldval - инкрементирую поле fieldno на value у кортежей со старым значением этого поля oldval
- DELETE fieldno value - удаляю кортежи, у которых поле fieldno равно value

Для транзакционности будет 3 стейтмента:

- BEGIN - начинаю транзакцию (не обязателен)
- COMMIT - коммичу транзакцию
- ABORT - откатываю транзакцию

### SELECT

Основная сложность SELECT - подчищение места за другими, кооперативный VACUUM.

Проблема с heap и undo в том, что старые версии все равно придется хранить на страницах, но in-place изменения мы делаем только тогда, когда места для новой версии достаточно (т.е. не меньше старого).
Поэтому это хранилище можно назвать гибридным: append only + delta.

При SeqScan проверка видимости происходит для каждого кортежа по отдельности по следующему правилу: для полученного кортежа выполняем UNDO пока не найдем видимую версию на ТЕКУЩЕМ месте, т.е. если был UPDATE, который этот кортеж поместил на это место, то мы прекращаем поиск и не следуем в другое место.

Операция:
Операция (возможно перед этим надо Lock взять, чтобы обновление никто не сделал, хотя хз):

1. Читаем очередной кортеж
2. Проверяем его видимость:
   1. Если выставлен CSN, то сравниваем с ним, иначе смотрим на XID - получаем его CSN:
      1. В прогрессе - CSN_IN_PROGRESS (спец. значение)
      2. Закоммичен - выставляем его CSN в поле
      3. Аборт - CSN_ABORTED (спец. значение)
   2. Если он не виден:
      1. Поочередно восстанавливаем кортеж из UNDO лога (применяем UNDO записи). Для каждого кортежа проверка видимости аналогична, за исключением того, что CSN поле обновлять не нужно.
      2. Если дошли до UPDATE, который переместил запись сюда, или INSERT, создавшего этот кортеж, то считаем, что кортежа не существовало
   3. Если кортежа нет, то пропускаем
3. Если tnx первой версии кортежа на странице в состоянии aborted:
   1. Применяем для этой записи UNDO, но только этой страницы (кооперативный vacuum)
4. Если после всех действий кортеж на странице не виден вообще:
   1. Если CSN кортежа меньше oldestCSN - удалить со страницы

### INSERT

Операция (возможно перед этим надо Lock взять, чтобы обновление никто не сделал, хотя хз):

1. Нахожу страницу с нужным местом
   1. Пока делаю это последовательно, без fsm
   2. Нужно дополнительно учитывать `vacated`, если не 0:
      1. Проходимся по всем itemid и находим те, у которых есть фрагментация (свободное место справа)
      2. Если tnx:
         1. завершилась - +space
         2. в процессе выполнения - не трогаем (с ней может случиться что угодно)
         3. аборт - +space (помогаем)
      3. После подсчета, если место норм, то выполняем дефрагментацию
2. X lock страницы
3. Делаю UNDO запись
4. Добавляю новую запись в line pointers
5. Вставляю кортеж на страницу
6. Lock release

Структура UNDO:

- Физическое расположение кортежа (pageno+itemid)

UNDO:

1. X lock страницы
2. Проверяю, что XID кортежа такой же как у меня (также UndoLocation мой), иначе ничего не делаю (ничего отменять не нужно, уже сделано (хотя мб мусор))
3. Помечаю запись в line pointers мертвой (можно без флагов, а спец. значения, например, длина 0)
4. Увеличиваю `vacated` на размер этого item
5. Release lock

### UPDATE

Операция:

Если можем перезаписать:

1. Делаем UNDO запись
2. Перезаписываем кортеж

Иначе:

1. Находим свободное место (возможно та же страница)
2. Создаем UNDO запись
3. Записываем новый кортеж
4. Старый кортеж помечаем удаленным
5. vacated увеличиваем на размер освободившихся байт кортежа

Структура UNDO:

- Данные старого кортежа (пока полная копия, пока без дельты)
- Расположение старого кортежа, (pageno+itemid)
- Расположение нового кортежа, (pageno+itemid) (возможно будет тем же самым)

UNDO:

1. Проверяю, что новый кортеж имеет мой XID + UndoLocation - иначе ничего не надо (запись применена, либо все сломалось)
2. Если ctid нового и старого совпадают, то значит был in-place update:
   1. Копирую старый кортеж на новое место
   2. vacated уменьшаем на разницу между новым и старым кортежем (если in-place update, то значит свободного места стало больше)
3. Иначе:
   1. Если новый кортеж существует (за меня никто его не удалил):
      1. Новый кортеж стираю
      2. itemid помечаю мертвым
      3. Увеличиваю nvacated на размер кортежа
   2. Если старый кортеж существует и его \[XID; UndoLocation\] мои:
      1. Копирую на старый кортеж его записи
      2. Уменьшаю vacated на размер кортежа

### DELETE

Операция:

1. Создаем Undo запись
2. Помечаем запись удаленной
3. vacated увеличиваем на размер кортежа

Структура UNDO:

- Данные старого кортежа (полная копия)

## Файловая организация

Весь код лежит внутри `src` директории.
Используются только `.cpp` и `.hpp` расширения.
Заголовочные файлы все лежат в `src/include`.
Если это публичный класс, то правило - файл на класс. Если это небольшая деталь реализации (вспомогательный класс), то можно в том же самом файле.
Название файла - название класса.
